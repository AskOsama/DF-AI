Constraint Satisfaction — Alibi Consistency on GPS Trajectories

The constraint-satisfaction section in Part 4 made the case on paper: a timeline is a CSP, and an alibi is contradicted when no assignment satisfies all the constraints at once. This page turns that story into a runnable pipeline over a real, open-access dataset — Microsoft's GeoLife GPS trajectories standing in for a seized phone's location log — and walks the whole chain from download to a court-ready finding.

Dataset & download

GeoLife GPS Trajectories (v1.3), Microsoft Research Asia. 17,621 real GPS trajectories from 182 users (2007–2012, mostly Beijing) — each a dense, time-stamped sequence of latitude/longitude points. It is exactly the shape of artefact a forensic examiner recovers from a phone or vehicle tracker: "where was this device at time T?"

Real data, constructed story. The GPS traces are genuine; the criminal context (suspect, CCTV, crime scene) is an invented teaching scenario layered on top — the honest way to get a realistic forensic exercise from an open dataset.

Forensic scenario

A suspect's phone is seized and its GPS log recovered (we use one GeoLife user's .plt file as the stand-in). The suspect claims an alibi: "From 14:00 to 15:00 on the incident date I was home in north Beijing — I was never near the crime scene in the south." A CCTV still places a person at the southern scene at 14:30. Is the alibi consistent with the phone's own data and the sighting, given that nobody travels faster than 120 km/h?

This is a consistency question, not an optimisation — we don't want the "best" timeline, we want to know whether any timeline can satisfy every fact at once. If none can, the alibi is contradicted by geometry, and we can name the exact constraint that broke it.

Encoding: data → CSP

Discretise the incident hour into ten-minute slots and ask the solver to place the device in regions consistent with both the recovered log and the claimed alibi.

CSP elementFrom the data
VariablesOne per time slot: P_1400, P_1410, … P_1500 (7 variables).
DomainsA small region grid built by binning lat/lon: {HOME_NORTH, MIDTOWN, CRIME_SCENE_SOUTH, ELSEWHERE}.
Anchor constraintsFor each slot the phone actually logged a point, pin it: e.g. P_1400 == HOME_NORTH.
Sighting constraintThe CCTV assertion under test: P_1430 == CRIME_SCENE_SOUTH.
Reachability constraintsBetween consecutive slots, reachable(a,b) holds only if haversine(a,b) ≤ v_max · Δt (Δt = 10 min, v_max = 120 km/h). HOME_NORTH→CRIME_SCENE_SOUTH (~25 km) is not reachable in one slot.

The CSP returns an assignment only if a physically possible timeline satisfies the anchors, the alibi, and the sighting. When it can't, that UNSAT result is the forensic finding.

Algorithm & data structures

  • Algorithm: backtracking search with constraint propagation (arc-consistency / forward checking). OR-Tools CP-SAT does this automatically and, crucially, returns a proof of infeasibility — a minimal conflicting set of constraints.
  • Constraint graph: nodes = time-slot variables; edges = binary reachability constraints between consecutive slots; unary constraints = anchors and the sighting. Stored as an adjacency list.
  • Region table: {region: (lat, lon)} plus a precomputed pairwise great-circle (haversine) distance matrix the reachability predicate consults.
  • Library: Google OR-Tools CP-SAT (ortools.sat.python.cp_model, Apache-2.0). For a transparent teaching version, python-constraint (BSD) exposes addVariable / addConstraint / getSolution directly.

Execution

  1. Pick one user folder as the seized phone; parse the .plt file (skip 6 header lines), filter to the incident date and the 14:00–15:00 window.
  2. Bin each logged point into a region; record, per slot, which region the phone was in → anchor constraints.
  3. Build the region centroid table and a haversine distance matrix.
  4. Build the CSP: slot variables over the region domain, unary anchors, the CCTV constraint at 14:30, binary reachability between consecutive slots.
  5. Solve and interpret: SAT → print the satisfying region-per-slot timeline; UNSAT → extract the minimal conflict set.
from ortools.sat.python import cp_model
from math import radians, sin, cos, asin, sqrt

REGIONS = ["HOME_NORTH", "MIDTOWN", "CRIME_SCENE_SOUTH", "ELSEWHERE"]
CENTROID = {"HOME_NORTH": (40.01, 116.32), "MIDTOWN": (39.95, 116.33),
            "CRIME_SCENE_SOUTH": (39.88, 116.33), "ELSEWHERE": (39.99, 116.50)}
SLOTS = ["1400", "1410", "1420", "1430", "1440", "1450", "1500"]

def km(a, b):                                   # great-circle distance
    (la1, lo1), (la2, lo2) = CENTROID[a], CENTROID[b]
    dla, dlo = radians(la2-la1), radians(lo2-lo1)
    h = sin(dla/2)**2 + cos(radians(la1))*cos(radians(la2))*sin(dlo/2)**2
    return 2*6371*asin(sqrt(h))

V_MAX, DT_H = 120, 10/60                         # km/h, hours per slot
m = cp_model.CpModel()
P = {s: m.NewIntVar(0, len(REGIONS)-1, f"P_{s}") for s in SLOTS}

# anchors from the recovered phone log (phone was HOME_NORTH all hour)
for s in SLOTS:
    m.Add(P[s] == REGIONS.index("HOME_NORTH"))
# the CCTV sighting under test
m.Add(P["1430"] == REGIONS.index("CRIME_SCENE_SOUTH"))
# reachability between consecutive slots
for a, b in zip(SLOTS, SLOTS[1:]):
    forbidden = [(i, j) for i in range(len(REGIONS)) for j in range(len(REGIONS))
                 if km(REGIONS[i], REGIONS[j]) > V_MAX*DT_H]
    m.AddForbiddenAssignments([P[a], P[b]], forbidden)

solver = cp_model.CpSolver()
status = solver.Solve(m)
print(solver.StatusName(status))                # -> INFEASIBLE

Final output

Result: INFEASIBLE — alibi inconsistent. No assignment of the device's position over 14:00–15:00 satisfies every fact at once. The phone log anchors it to HOME_NORTH at 14:30, while the CCTV sighting requires CRIME_SCENE_SOUTH at 14:30; the two regions are ~25 km apart and cannot be occupied in the same slot. Minimal conflict set: {phone-anchor@14:30, CCTV@14:30}. The alibi as stated cannot be true.

For contrast, move the CCTV time to 15:30 and the solver returns a satisfying timeline (HOME_NORTH → MIDTOWN → CRIME_SCENE_SOUTH) — the SAT case, showing the method does not simply always cry "guilty". The value for court is that the answer is a concrete assignment (or a named contradiction), not a probability.

Runnable example & verified output: the script, the real captured output, and the plan/execute prompts are in examples/01-csp-geolife/. The verified run on a real 1,116-point GeoLife trajectory returns INFEASIBLE with the CCTV-vs-phone-log contradiction as its UNSAT core.
