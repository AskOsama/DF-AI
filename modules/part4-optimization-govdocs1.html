Optimization & Trade-offs — Triage Prioritisation on a File Corpus

The optimization section framed triage as maximising "expected evidence per analyst-hour", with a Pareto front of defensible compromises rather than a single best answer. This page makes that concrete over the open-access GovDocs1 corpus: 1,000 carved files, two examiners, a court deadline — and an optimiser that beats greedy, ε-greedy and random ordering.

Dataset & download

GovDocs1 (Digital Corpora). Roughly one million real files (PDF, DOC, XLS, PPT, HTML, JPG, TXT, CSV…) harvested from .gov servers — the de-facto standard test corpus in academic digital forensics. Conveniently shipped as subset0.zip … subset9.zip, each 1,000 files, so a student can pilot on a single subset.

Forensic scenario

A drive image from a seized workstation has been carved, yielding the 1,000 files of one GovDocs1 subset (treat each as one carved artefact). Two examiners have a combined 16 analyst-hours before a court deadline. In what order, and on whose queue, should files be examined to maximise expected evidentiary value within the time budget? Spreadsheets and documents are high-value-per-minute; large media are slow; a few malware-flagged files are high-value but expensive.

Encoding: files → items

Each carved file becomes a record with a transparent cost and value:

AttributeHow it's computed (cheaply, reproducibly)
cost_minutesbase[ext] + k · size_MB, e.g. base = {xls:8, doc:6, pdf:5, ppt:5, html:3, jpg:4, txt:2}, k ≈ 2.
value_scorefile-type prior + malware-scan-report bonus + regex/entropy hits (emails, card/IBAN patterns, high-entropy = possible encryption/stego).
item tuple(id, ext, size, cost_minutes, value_score)
2nd objective (Pareto)coverage = number of distinct file types touched — thoroughness traded against value and time.

Keep the value heuristic disclosable. Every weight (type priors, the malware bonus, the regex set) is a number you can defend in court. The moment "value" becomes a black-box score, the triage order stops being explainable — the same trap Part 4 warns about for ML.

Algorithm & data structures

  • Baselines (from Part 4): greedy (sort by value/cost), ε-greedy, random, and brute-force (only on a ≤20-item slice) — the strategies to beat.
  • Single-objective knapsack: capacity = analyst-minutes, maximise Σ value. OR-Tools KnapsackSolver; data: NumPy value/cost vectors, a heapq priority queue for the greedy baseline.
  • Two examiners: a multiple-knapsack / CP-SAT model assigns files to two 8-hour bins, maximising total value within each budget.
  • Multi-objective Pareto: DEAP NSGA-II (tools.selNSGA2, tools.ParetoFront) over objectives (maximise value, maximise coverage, minimise time); an Individual is a binary include/exclude vector.

Execution

  1. Download one subset and unzip into ./corpus.
  2. Featurize: walk the corpus, compute cost_minutes and value_score per file → items.csv.
  3. Run the four baselines for a single 16-hour budget; record total value.
  4. Solve the scalar optimum (one budget) and the two-examiner split (CP-SAT); compare against baselines.
  5. Run NSGA-II for the value/coverage/time Pareto front; collect the non-dominated plans.
import numpy as np
from ortools.algorithms.python import knapsack_solver

# items.csv already has cost_minutes (int) and value_score (int) per file
values  = items["value_score"].to_list()
weights = [items["cost_minutes"].to_list()]      # one dimension: minutes
capacity = [16*60]                                # 16 analyst-hours

s = knapsack_solver.KnapsackSolver(
    knapsack_solver.SolverType.KNAPSACK_MULTIDIMENSION_BRANCH_AND_BOUND_SOLVER,
    "triage")
s.init(values, weights, capacity)
best = s.solve()
chosen = [i for i in range(len(values)) if s.best_solution_contains(i)]
print(f"selected {len(chosen)} files, total value {best}")

# greedy baseline for comparison
order = sorted(range(len(values)),
               key=lambda i: values[i]/weights[0][i], reverse=True)

Final output

The analyst receives two things:

  • A ranked examination order split into Examiner A and Examiner B queues, each respecting its 8-hour budget, with the total expected-evidence score and a table showing the optimiser beating greedy / ε-greedy / random.
  • A Pareto front of 4–5 defensible alternative plans — e.g. "high-value / narrow" vs. "broad-coverage / lower-value" — so the lead can pick the trade-off that fits the case.
Examiner A (480 min):  142 files,  value 318.7   [docs+sheets first]
Examiner B (471 min):  120 files,  value 296.1   [malware-flagged + PDFs]
vs greedy 571.2 | optimiser 614.8 | random 402.5

Pareto front (value, types covered, minutes):
  P1  614.8   6   951      <- max value
  P3  571.0   9   948
  P5  498.4  11   930      <- max coverage

Runnable example & verified output: the script, the real captured output, and the plan/execute prompts are in examples/02-optimization-govdocs1/. On 981 real carved files the optimiser beats random ≈5.6× and produces a 121-point NSGA-II Pareto front.
