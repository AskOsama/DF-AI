Feature Engineering — the Agent's Run on Real PE Files

The executed outcome behind the worked prompt for feature engineering. The pipeline really trained a classifier on EMBER-style features extracted from real Windows binaries; below is the agent's loop — Approach → Action → Observation → Done — with the genuine captured output, including a real leakage-trap demonstration. The feature families and model live on the Part 4 dataset page.

What was run

Forensic question: an unknown executable is carved from unallocated space. Is it benign or packed/suspicious — decided without running it, from static byte/header features? The agent builds an EMBER-style feature matrix from real benign Windows PEs plus clearly-labelled synthetic packed twins, trains LightGBM, scores an unknown, and then demonstrates the leakage trap for real. Verified outcome: honest accuracy 0.92 / AUC 0.97; a leaky feature inflates accuracy to a perfect 1.0 — the cautionary tale.

Plan-phase prompt

What the analyst types to make the agent design the classifier (no code yet):

You are a forensic ML engineer. PLAN (do not write final code yet) a static
malware/file-triage classifier in the EMBER style. Constraints:
  - SAFETY: never download or execute live malware. Use either EMBER's
    pre-extracted numeric vectors, or features from real benign PE files plus
    clearly-labelled synthetic "packed" samples.
  - Decide benign vs packed/suspicious WITHOUT running files.
Deliver a plan covering:
  1. Data sourcing: which safe corpus, how labelled, license + safety justification.
  2. Feature engineering: exact families (byte histogram, windowed Shannon
     entropy, printable-string stats, PE header fields, per-section entropy,
     import/DLL counts) and how each is computed from raw bytes.
  3. Matrix design: row = file, columns = named features, label vector.
  4. Model (LightGBM GBDT; RandomForest fallback) and why no scaling.
  5. Evaluation: stratified split, accuracy + ROC-AUC + report, importances,
     per-file driver explanation.
  6. A leakage-trap experiment: name >=2 leaky features (source-folder id, PE
     timestamp); measure the inflation, then prove the honest number after removal.
  7. Risks and what "good enough for triage" means (lead, not verdict).

Execute-phase prompt

What the analyst types to run it:

Implement the approved plan as a single runnable run.py with NO required network
access (build the corpus from local System32 PE files + synthetic packed twins).
Print, in order: corpus summary; feature-matrix shape + class means proving the
features separate the classes; train/test sizes; accuracy + ROC-AUC +
classification report; top-8 named feature importances; a scored held-out unknown
with verdict + probability + top-3 drivers; and the leakage demo (5-fold CV
accuracy & AUC WITH vs WITHOUT the leaky feature, plus its importance rank).
Fall back to RandomForest if lightgbm import fails, and to byte-only features if
lief is unavailable. Run with `python3 run.py | tee output.txt` and report the
REAL numbers — never fabricate.

The run: Approach → Action → Observation

Each Observation is verbatim from the real output.txt.

Step 1 — Gather a safe labelled corpus

Approach. Never touch live malware. Use real benign PEs + clearly-labelled synthetic packed twins.

Action:  gather_corpus()

Observation:
    benign (REAL System32 PE) : 60
    packed (SYNTHETIC)        : 60
    PE parser (lief): available    classifier: LightGBM

Step 2 — Extract EMBER-style features (raw bytes → numbers)

Approach. Turn each file into a fixed numeric vector; confirm the engineered features actually separate the classes.

Action:  extract_features() -> matrix

Observation:
    feature matrix X : shape (120, 31)
    families: byte-histogram(16), entropy summary(5), strings(2), PE hdr/sect/imports(7)
    mean whole-file entropy  benign=4.659  packed=5.495   (separation visible)

Step 3 — Train and evaluate honestly

Approach. Stratified split, train LightGBM, report accuracy + ROC-AUC + the full classification report.

Action:  train_test_split(); LGBMClassifier.fit(); evaluate()

Observation:
    train 84 / test 36
    HONEST test accuracy : 0.9167
    HONEST test ROC-AUC  : 0.9722
                  precision    recall  f1   support
        benign      1.000     0.833  0.909    18
        packed      0.857     1.000  0.923    18

Step 4 — Explain the drivers + score an unknown

Approach. Surface the top features for a human reviewer, then score one held-out unknown with its drivers.

Action:  feature_importance(); score_unknown("packed_043.bin")

Observation:
    top importances: byte_hist[9], pe_section_entropy_max, byte_entropy_win_max,
                     byte_entropy_win_std, pe_timestamp ...
    unknown packed_043.bin -> PACKED/SUSPICIOUS  P(packed)=1.000  (truth: packed)
      drivers: pe_section_entropy_max(+65.79), byte_hist[9](+65.13), byte_entropy_win_std(+57.93)

Step 5 — The leakage trap (real demonstration)

Approach. Add a leaky feature (source_folder_id: benign=0, packed=1 — the "folder named evidence" shortcut) and compare honest vs leaky with 5-fold CV.

Action:  cross_val(honest); cross_val(with_leaky_feature)

Observation:
    WITHOUT leaky feat. (honest): accuracy=0.9417  AUC=0.9889
    WITH leaky feature          : accuracy=1.0000  AUC=1.0000
       -> leaky feature ranked #1 of 32 in importance (it dominates)
    INFLATION from leakage      : accuracy +0.0583   AUC +0.0111

>>> Human-in-the-loop gate. The verdict is a lead, not a verdict — it ranks files for an analyst to confirm. And the agent must exclude leaky metadata (source_folder_id, pe_timestamp) from the scoring model.

Done — the result

Honest model : accuracy 0.92 / AUC 0.97   (drivers = section & byte entropy)
Unknown scored: PACKED, P=1.000, driven by high-entropy sections (crypter-like)
Leakage trap : a folder-id feature -> accuracy 1.0000 / AUC 1.0000
   LESSON: the leaky model learned the FOLDER, not the FILE. On genuinely
   unsorted evidence the shortcut is gone and it collapses
   -> drop folder/timestamp/source-feed metadata before training.

The headline for class: engineered features (section/byte entropy) genuinely separate packed from benign, and the leakage demo shows why a too-good number is a red flag — the exact "feature that encodes the answer" warning, measured in real percentages.

Guidance: reading the outcome

  • A lead, not a verdict. 0.92 accuracy ranks files for human review; it does not convict a file. Pair every score with the named drivers.
  • Distrust the perfect score. The leaky model's 1.0 is the tell: it learned the intake folder, not the file. On unsorted evidence it collapses. Exclude folder/timestamp/source-feed features before training.
  • Failure mode — unparseable files. If the PE parser returns nothing, the agent must not silently emit zeros and call it benign (zeroed structural features bias toward "benign / no imports"). It should flag "unparseable — manual review" and fall back to byte-only features with reduced confidence.

Honest deviation (teaching note). The full multi-GB EMBER benchmark and its fragile ember pip package were deliberately skipped for a safe, reproducible local demo. The benign class is 60 real signed System32 PEs; the "packed" class is 60 synthetic twins (partially XOR-encrypted + high-entropy blobs), never executed. The same feature code is what EMBER applies at million-sample scale.
