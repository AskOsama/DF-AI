What was run

Forensic question: Subject X claims to be in Dubai; last confirmed at London Heathrow (LHR); the crime scene is Bogotá (BOG). Could X have reached BOG from LHR within ≤3 connections — and if so, via which hub and which real airlines? The agent loads a flight knowledge base, tests reachability, and surfaces an auditable derivation. Verified outcome: no direct LHR→BOG flight, but BOG is reachable in one connection via Atlanta (Delta); two independent engines agree.

Plan-phase prompt

What the analyst types to make the agent plan the inference (no execution yet):

Role: forensic-reasoning planner.
Goal: PLAN (do not execute) a knowledge-representation pipeline that decides
whether a suspect could have travelled from one airport to another within a
bounded number of connections, using the OpenFlights route database.

Produce a numbered plan that:
  1. States the question precisely (LHR -> BOG, <= 3 connections = <= 4 legs)
     and what a positive/negative answer implies for the Dubai alibi.
  2. Identifies the data (routes.dat, airports.dat), columns, license (ODbL),
     snapshot date, and how nulls (\N) are handled.
  3. Specifies the FACT schema (airport/3, flight/3) and the RULES (reachable/2
     base + recursive with a cycle guard; contradicts_alibi/1) in Datalog/Prolog.
  4. Chooses data structures (directed multigraph) + reasoning strategy
     (backward vs forward chaining) and justifies BOUNDING the search.
  5. Defines how to extract an auditable derivation chain (GOAL/RULE/FACT tree)
     and how to cross-validate two independent engines.
  6. Lists verification checks (direct-edge check, both legs present, engine
     agreement) and stop conditions.
Output only the plan and its assumptions/risks; write no code yet.

Execute-phase prompt

What the analyst types to run it:

Implement run.py and run it on the real OpenFlights data.
  - Parse airports.dat -> airport(IATA,City,Country) (skip invalid IATA);
    routes.dat -> flight(Airline,Src,Dst) (skip \N). Print fact/node/edge counts.
  - Build a networkx.MultiDiGraph (airport attrs = city/country; edge = airline).
  - Answer reachability LHR -> BOG within <= 4 legs: print whether a direct edge
    exists; prove existence with bounded BFS; enumerate short paths with
    all_simple_paths(cutoff=3); show one 1-connection path with the REAL airlines
    on each leg and VERIFY both legs exist; report the hub with city/country.
  - Print an auditable backward-chaining derivation chain (GOAL -> RULE -> FACT).
  - If pyDatalog imports: assert flight/2 facts for the bounded candidate
    subgraph, define recursive reachable/2, query reachable('LHR','BOG'), and
    CROSS-CHECK it matches networkx; else document and rely on networkx.
  - Save real output: python3 run.py | tee output.txt. Do not fabricate numbers.

The run: Approach → Action → Observation

Each Observation is verbatim from the real output.txt.

Step 1 — Build the knowledge base

Approach. Parse the route/airport facts into a directed multigraph so reachability becomes a graph query.

Action:  parse_facts(); build_graph()

Observation:
    airport/3 facts parsed :   6072  (IATA -> City, Country)
    flight/3  facts parsed :  66934  (Airline, Src, Dst)
    LHR = London, United Kingdom    BOG = Bogota, Colombia

Step 2 — Cheapest disqualifier: a direct flight?

Approach. Before any search, check whether a direct LHR→BOG edge exists.

Action:  query_kb(flight, src='LHR', dst='BOG')

Observation:
    Direct flight LHR -> BOG exists? False

Step 3 — Is BOG reachable within the legal bound?

Approach. No direct flight — ask the reasoner whether BOG is reachable from LHR within 4 legs (≤3 connections).

Action:  query_kb(reachable, 'LHR', 'BOG', max_legs=4)

Observation:
    Bounded BFS: reachable within 4 legs? True
    1-connection (2-leg) hubs LHR->hub->BOG: 16
    Simple paths <= 3 legs: 35980  (209 are 1-connection)

Step 4 — Surface a concrete, verifiable route

Approach. Reachable — I need a real route to show. Find a 1-connection hub and confirm both legs exist with their operating airlines.

Action:  find_hubs(src='LHR', dst='BOG'); query_kb(flight, ...)

Observation:
    route : LHR -> ATL -> BOG   (London -> Atlanta -> Bogota)
      leg LHR->ATL  airlines = AA, AF, AY, BA, DL, IB, KL, VS   edge_exists=True
      leg ATL->BOG  airlines = DL                               edge_exists=True
    HUB: ATL (Atlanta, United States)

Step 5 — Auditable derivation + cross-check

Approach. Build the backward-chaining proof tree and cross-check with a second engine (pyDatalog) before reporting.

Action:  derive_chain('LHR','BOG'); crosscheck(networkx, pyDatalog)

Observation:
    GOAL: reachable('LHR','BOG')
      RULE: reachable('LHR','BOG') :- flight(_,'LHR','ATL'), reachable('ATL','BOG').
      |- flight('AA','LHR','ATL')          [FACT from routes.dat]
      |- subgoal reachable('ATL','BOG')
         |- flight('DL','ATL','BOG')       [FACT from routes.dat]
    pyDatalog reachable('LHR','BOG') = True ;  networkx = True
    CROSS-CHECK: pyDatalog == networkx ?  True

>>> Human-in-the-loop gate. The agent never records a conclusion. A positive reachability result is an investigative lead requiring analyst sign-off before it enters a case file. Reachability ≠ proof of travel: it shows X could have flown the route, not that X did.

Done — the result

DERIVED: reachable('LHR','BOG') = TRUE  via  LHR --AA--> ATL --DL--> BOG
   - no direct flight; 1 connection; hub = Atlanta (ATL)
   - every edge is a named, real airline service from routes.dat
   - two independent engines agree (networkx == pyDatalog)
=> The "Dubai" alibi does NOT make BOG physically unreachable.

The pedagogical payoff: the engine outputs not a probability but a path whose every edge is a named real airline service, with a derivation that cites the exact rule applied at each step — an auditable chain an opposing expert can read line by line.

Guidance: reading the outcome

  • Reachable ≠ travelled. The result is a lead: X could have flown LHR→ATL→BOG. It does not prove X did, and it requires human sign-off.
  • The chain is the value. Because every step traces to a fact/rule, the conclusion is auditable — the whole reason to prefer rule-based reasoning over a black box for evidence work.
  • Failure mode — snapshot completeness. The conclusion is only as complete as the route snapshot (here ~June 2014). Routes added later (e.g. a new direct LHR→BOG) or seasonal/cancelled routes are invisible, so the model can both miss real options and assert routes no longer flown. Codeshares, minimum connection times, visas, and seat availability are not modelled. Always cite the snapshot date.

Honest deviation (teaching note). The flight graph is real; the suspect/alibi framing is a constructed teaching scenario. pyDatalog runs on a bounded candidate subgraph (1,430 deduped edges among 185 airports on ≤3-leg paths) because naïve recursion over all 66,934 edges blows up; networkx remains authoritative over the full graph, and the two agree.
