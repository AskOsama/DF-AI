Feature Engineering — Static Malware Triage from Raw Bytes

The data-preparation section made the core point: a file is not handed to a classifier as a file — it is handed as a list of features (size, entropy, header bytes, …), and poor features sink even a perfect algorithm. EMBER is the canonical example of exactly that: a Windows executable reduced to a 2,380-dimensional engineered vector. This page walks the raw-bytes → features → verdict pipeline, including the leakage trap to avoid.

Dataset & download

EMBER 2018 (v2 features) — Elastic Malware Benchmark for Empowering Researchers. Engineered static features extracted from ~1 million Windows PE files (malicious / benign / unlabeled), released with the extraction source code — so students can see exactly how raw bytes become features. The textbook "raw artifact → feature vector" story.

Forensic scenario

An analyst carves an unknown Windows executable from unallocated space — no signature, no AV verdict. Is it benign or malware? — answered without running it, purely from measurable byte-level and structural properties. At scale, the same scoring ranks 10,000 carved executables so a human reviews the most suspicious first.

Encoding: the feature vector

Each raw PE file becomes a fixed 2,380-dimensional vector across nine groups (feature version 2):

Group (dims)Computed from the raw bytes
Byte histogram (256)Count of each byte value 0x00–0xFF, normalised — gross composition.
Byte-entropy histogram (256)Sliding-window Shannon entropy; high regions flag packing/encryption.
String stats (109)Printable-string count, mean length, char-distribution entropy, path/URL/registry markers.
General info (10)File size, virtual size, imports/exports counts, has-debug/TLS/signature.
Header info (62)COFF machine type/magic, timestamp, characteristics; subsystem, linker version.
Section info (305)Per-section name, size, entropy, access flags — feature-hashed.
Imports (1,280)DLLs + API functions (e.g. VirtualAlloc, WriteProcessMemory) — hashed.
Exports (128)Exported function names — feature-hashed.
Data directories (30)Size + address of the 15 PE data directories.

Leakage trap. The PE header timestamp leaks under EMBER's time-split (train before a cutoff, test after): a model can learn "newer timestamp ⇒ class X", a shortcut that won't generalise to genuinely unknown files. Same risk for any filename/path/source-feed metadata — the exact "folder named evidence" shortcut Part 4 warns about. Drop or scrutinise timestamp-derived features.

Algorithm & data structures

  • Feature matrix: one row per file = the 2,380 floats above, concatenated in fixed group order; the corpus is X of shape (n, 2380) (float32 memmap) and labels y (0 benign, 1 malware, -1 unlabeled → dropped).
  • Classifier: LightGBM gradient-boosted trees (MIT, LGBMClassifier) — the official EMBER baseline (AUC ≈ 0.999). GBDTs handle heterogeneous, unscaled, hashed features natively, no normalisation needed. Alternative: scikit-learn RandomForestClassifier.
  • Explainability: model.feature_importance() mapped back to the nine groups; per-sample SHAP for the one artefact under review.

Execution

  1. Download & extract EMBER; pip install lightgbm and the ember helper package.
  2. Vectorize the JSONL raw features → X_*.dat / y_*.dat (this step is the feature-engineering demo).
  3. Load and drop unlabeled rows (y == -1).
  4. Train LightGBM; print feature importances mapped to the groups.
  5. Leakage check: retrain excluding header-timestamp features; compare test AUC over the time-split.
  6. Classify the carved unknown; explain the single sample with SHAP.
import ember
ember.create_vectorized_features("ember2018/")          # raw JSONL -> matrix
X, y = ember.read_vectorized_features("ember2018/", "train")
X, y = X[y != -1], y[y != -1]                            # drop unlabeled

model = ember.train_model("ember2018/")                  # LightGBM baseline

# classify the carved unknown, statically (never executed)
with open("sample_4471.exe", "rb") as f:
    score = ember.predict_sample(model, f.read())
print(f"malware probability = {score:.3f}")              # -> 0.987

Final output

The analyst sees a verdict, a confidence, and — most importantly for court — the features that drove it:

sample_4471.exe
  VERDICT:     MALWARE
  CONFIDENCE:  0.987
  TOP DRIVERS (feature importance / SHAP):
    1. Section entropy (.text) = 7.92        -> packing
    2. Imports: VirtualAlloc, WriteProcessMemory -> process injection
    3. Byte-entropy histogram high bins
    4. Low printable-string count            -> packed
  NOTE: timestamp features excluded to avoid temporal leakage.

As Part 4 stresses, this is a lead to verify, not a verdict — the feature attribution gives the human reviewer a defensible starting point, and the leakage note keeps the method honest.

Runnable example & verified output: the script, the real captured output, and the plan/execute prompts are in examples/04-feature-engineering-ember/. LightGBM scores accuracy 0.92 / AUC 0.97 on real PE features, and the leakage demo inflates accuracy to a perfect 1.0 — exactly the trap to teach.
