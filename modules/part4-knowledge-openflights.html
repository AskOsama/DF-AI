Knowledge & Reasoning — Path Inference over a Flight Network

The knowledge-reasoning section argued that the forensic payoff of rule-based systems is transparency: every conclusion traces back to the exact rules and facts that produced it. This page demonstrates that over the open-access OpenFlights route graph — deriving whether a suspect could have reached a city by connecting flights, and citing each real airline service in the chain.

Dataset & download

OpenFlights Routes & Airports. A global graph of ~67,663 scheduled airline routes between ~3,321 airports on 548 airlines, plus a gazetteer of 10,000+ airports with city/country/geo metadata. Tiny, plain-CSV, and trivial to load entirely in memory.

Real data, constructed story. The route graph is genuine (a fixed June-2014 snapshot — perfect because every student gets identical results). The suspect/alibi framing is an invented teaching scenario, exactly the kind of made-up case the brief invites when no off-the-shelf forensic dataset fits.

Forensic scenario

Subject X claims they were in Dubai on the night of an incident and therefore could not have been in Bogotá (BOG), where the crime occurred. Border records place X departing London Heathrow (LHR) ~28 hours earlier, and there is no direct LHR→BOG service in the route graph. Is there any chain of connecting flights (≤3 connections) that would let X travel LHR → BOG within the window — and through which hubs? A "yes, via hub H" contradicts the alibi, and the engine names the exact flights that make it possible.

Encoding: facts & rules

Each CSV row becomes a ground fact; reachability is defined by rules.

# Facts extracted from the CSVs
airport('LHR', 'London', 'United Kingdom').
airport('BOG', 'Bogota', 'Colombia').
flight('BA', 'LHR', 'MAD').        % from routes.dat
flight('AV', 'MAD', 'BOG').

# Rules (the inference engine)
reachable(A, B, [A,B])      :- flight(_, A, B).               % one hop
reachable(A, C, [A|Path])   :- flight(_, A, B),               % transitive
                               reachable(B, C, Path),
                               \+ member(A, Path).             % no cycles
contradicts_alibi(S, O, C, Path) :- claimed_origin(S, O),
                                     reachable(O, C, Path),
                                     length(Path, L), L =< 4.  % ≤3 connections

The same idea in pyDatalog (the primary library), with facts loaded in a loop from routes.dat:

from pyDatalog import pyDatalog
pyDatalog.create_terms('flight, reachable, X, Y, Z, W')
+ flight('BA', 'LHR', 'MAD'); + flight('AV', 'MAD', 'BOG')
reachable(X, Y) <= flight(Z, X, Y)                  # base case
reachable(X, Y) <= flight(Z, X, W) & reachable(W, Y) # recursive case
print(reachable('LHR', 'BOG'))                       # query

Algorithm & data structures

  • Inference strategy: backward chaining (goal-directed: "is BOG reachable from LHR?") — the natural fit, exploring only routes relevant to the goal. A forward-chaining variant (materialise the full transitive closure, then look up) is a good contrast exercise.
  • Knowledge graph: a directed multigraph — nodes = airports (IATA), edges = flights (one edge per airline service). Concretely a networkx.MultiDiGraph; routes.dat rows map 1:1 to edges, airports.dat supplies node attributes.
  • Reasoning ≡ search: recursive reachability with cycle-avoidance is bounded DFS/BFS; cross-check with networkx.all_simple_paths(G, src, dst, cutoff=3) and has_path.
  • Libraries: pyDatalog (LGPL) for the rule engine; networkx (BSD) for the graph + path algorithms; optional SWI-Prolog for a pure-Prolog version, or Experta/CLIPS for a forward-chaining RETE demo.

Execution

  1. Download routes.dat and airports.dat.
  2. Parse facts: airports → airport(IATA, City, Country); routes → flight(Airline, Src, Dst), skipping rows with \N codes.
  3. Assert all flight/3 facts into pyDatalog (and/or build the MultiDiGraph).
  4. Define the reachable rules and the hop budget (≤3 connections).
  5. Query reachable('LHR','BOG'); extract the witnessing path and the flight/3 facts on each edge.
  6. Cross-validate with networkx.all_simple_paths(G,'LHR','BOG',cutoff=3).
import csv, networkx as nx

G = nx.MultiDiGraph()
with open('data/airports.dat', encoding='utf-8') as f:
    for r in csv.reader(f):
        iata = r[4]
        if iata != '\\N':
            G.add_node(iata, city=r[2], country=r[3])
with open('data/routes.dat', encoding='utf-8') as f:
    for r in csv.reader(f):
        src, dst, airline = r[2], r[4], r[0]
        if src in G and dst in G:
            G.add_edge(src, dst, airline=airline)

paths = list(nx.all_simple_paths(G, 'LHR', 'BOG', cutoff=3))
print(paths[0])     # -> ['LHR', 'MAD', 'BOG']

Final output

Derived fact: contradicts_alibi('SubjectX','LHR','BOG',['LHR','MAD','BOG']) — TRUE. A one-connection route via Madrid exists, well within the time window, so the Dubai alibi is inconsistent with the flight network.

The pedagogical payoff is the auditable chain — not a probability, but a path whose every edge is a named real airline service and whose derivation cites the recursive rule applied at each step:

GOAL: reachable('LHR','BOG')
 ├─ RULE  reachable(X,Y) <= flight(_,X,W) & reachable(W,Y)
 │   ├─ FACT  flight('BA','LHR','MAD')     [routes.dat: BA,…,LHR,…,MAD]
 │   └─ GOAL  reachable('MAD','BOG')
 │        └─ RULE reachable(X,Y) <= flight(_,X,Y)
 │             └─ FACT flight('AV','MAD','BOG')  [routes.dat: AV,…,MAD,…,BOG]
 └─ DERIVED PATH: LHR --BA--> MAD --AV--> BOG   (connections = 1 ≤ 3)

Over 66,934 real routes the run finds no direct LHR→BOG but a 1-stop path via Atlanta (Delta), with an auditable fact/rule chain (pyDatalog ↔ networkx cross-checked).
