Search Engines — Inverted Index & Ranking for E-Discovery

The search section made the point that underneath much of forensics is plain search: build an index once, then retrieve fast. This page builds a real inverted index over the Enron email corpus — the classic e-discovery dataset — and runs ranked keyword queries that surface the most probative messages first instead of scanning half a million files linearly.

Dataset & download

Enron Email Dataset (CMU distribution). ~500,000 real corporate emails from ~150 senior Enron employees, released by FERC during the 2001–2004 investigation — the canonical forensic / e-discovery corpus, and the standard testbed for information-retrieval research.

PropertyValue
LicensePublic, freely downloadable for research; CMU asks only that users respect the privacy of those involved. (No formal SPDX licence — treat as research-public.)
Size~1.7 GB gzipped; ~2.6 GB uncompressed; ~0.5M messages.
FormatMaildir tree — one RFC-822 text file per message (maildir/<user>/<folder>/<n>) with standard headers + body.
Homecs.cmu.edu/~enron (May 2015 version)
 
wget https://www.cs.cmu.edu/~enron/enron_mail_20150507.tar.gz
tar -xzf enron_mail_20150507.tar.gz          # creates ./maildir/
pip install rank-bm25 nltk                    # BM25 ranking + tokeniser
python -m nltk.downloader stopwords punkt
# persistent on-disk index alternative: pip install Whoosh

Scaling note. 2.6 GB is the artefact a student actually indexes; the same code scales to the multi-terabyte seized-disk collection Part 4 describes — the only change is sharding the index. (For a literal seized-disk story instead, Digital Corpora's M57-Patents scenario is the alternative.)

Forensic scenario

An e-discovery analyst under a litigation hold must surface every internal email discussing off-balance-sheet special-purpose entities and the Raptor/LJM hedging vehicles, ranked by relevance, so counsel reviews the most probative documents first. Rather than linearly scanning 500k files, the analyst builds an inverted index once, then issues sub-second ranked queries (raptor, LJM, "special purpose entity") and pivots on the top hits.

Encoding: the inverted index

Raw RFC-822 messages become an inverted index in four passes:

PassWhat happens
Parseemail.parser reads each maildir file; keep Message-ID as the doc ID, retain From/To/Subject/Date as fields + the body as the searchable content.
TokenizeSplit body+subject on non-alphanumeric boundaries (\w+); optionally keep email-address tokens intact.
NormalizeLowercase → drop stopwords → Porter/Snowball stem (hedging→hedg) → drop <2-char tokens.
Postingsterm → [(doc_id, term_freq, [positions])]; positions enable phrase/proximity queries. A parallel doc_store maps doc_id → {subject, from, date, snippet} for display.

Ranking model: BM25 (Okapi) as the primary scorer (k1=1.5, b=0.75), with TF-IDF as the simpler baseline. BM25 is preferred because it saturates term frequency and length-normalises — which matters for the wildly varying email lengths in the corpus.

Algorithm & data structures

  • Inverted index: dict[str, list[Posting]] (term → postings list) plus a document-length table and corpus average length for BM25.
  • Scoring: sum BM25 over query terms; rank docs by descending score. The Part-4 search strategies map onto retrieval — brute-force = linear scan of all 500k files (the thing we avoid); greedy = expand the top-scored postings first; ε-greedy/random = occasionally sample lower-ranked hits for diversity during review.
  • Libraries: rank_bm25 (Apache-2.0) for a transparent from-scratch pipeline; scikit-learn TfidfVectorizer for the baseline; Whoosh for a real on-disk index with a query parser; Tantivy / Elasticsearch for production scale-out.

Execution

  1. Download & extract the corpus; install rank-bm25 + nltk.
  2. Ingest: walk maildir/, parse each file, build doc_store[doc_id] = {from,to,subject,date,body}.
  3. Preprocess: lowercase → tokenize → drop stopwords → stem → tokenized_corpus.
  4. Build the index: BM25Okapi(tokenized_corpus) (or a Whoosh on-disk schema).
  5. Query: preprocess the query the same way → get_scores → argsort → top-k.
  6. Display: join top-k IDs back to doc_store for sender/date/subject + a keyword-in-context snippet; refine with phrase constraints and from: filters.
import os, email
from rank_bm25 import BM25Okapi
from nltk.corpus import stopwords
from nltk.stem import SnowballStemmer

stop, stem = set(stopwords.words('english')), SnowballStemmer('english')
def prep(text):
    return [stem.stem(w) for w in __import__('re').findall(r'\w+', text.lower())
            if w not in stop and len(w) > 1]

docs, corpus = [], []
for root, _, files in os.walk('maildir'):
    for fn in files:
        msg = email.message_from_file(open(os.path.join(root, fn),
                                           errors='ignore'))
        body = msg.get_payload() if isinstance(msg.get_payload(), str) else ''
        docs.append({'from': msg['From'], 'date': msg['Date'],
                     'subject': msg['Subject'], 'body': body})
        corpus.append(prep((msg['Subject'] or '') + ' ' + body))

bm25 = BM25Okapi(corpus)
scores = bm25.get_scores(prep('raptor LJM special purpose entity'))
top = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:10]

Final output

A ranked hit list the analyst reviews top-down — each row showing relevance score, structured metadata, and a keyword-in-context snippet:

The deliverable demonstrates the core forensic claim: build an index once, then retrieve ranked hits in milliseconds — letting counsel review the most probative documents first instead of scanning 500k files linearly.
