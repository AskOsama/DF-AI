Optimization — the Agent's Run on 981 Real Carved Files

What was run

Forensic question: a seized drive yields 981 carved files; an examiner has a 16-hour (960-minute) review budget. Which files to review first to maximise investigative value within the budget — and how to split the work across two examiners? The agent featurizes every file (cost, value), races baselines against a provably-optimal knapsack, and produces a Pareto front of trade-offs. Verified outcome: optimised triage ≈ 5.6× the random baseline; the knapsack strictly beats greedy at tight budgets; a 121-point NSGA-II Pareto front frames the human's choice.

Plan-phase prompt

What the analyst types to make the agent plan the triage (no execution yet):

You are a digital-forensics triage planner. A seized drive's files are extracted
to data/corpus/. An examiner has a 16-hour budget (960 min). Produce a PLAN (do
not execute) to decide which files to review first to maximise investigative
value within budget.

In your plan:
  1. How you estimate a review COST in minutes from each file's type and size
     (state the per-type base costs).
  2. A transparent VALUE score = type prior + PII/keyword bonus (emails, IBANs,
     Luhn-valid cards via byte sampling) + entropy term. List every weight — it
     must be court-defensible, no hidden tuning.
  3. The optimisation models: 0/1 knapsack (single budget), multiple-knapsack /
     CP-SAT split for two examiners (2 x 480 min), NSGA-II for value / coverage /
     time trade-off.
  4. Baselines (greedy density, random, brute-force slice) and the acceptance
     check (optimiser >= greedy).
  5. Risks: the value model is a proxy — where could it mis-prioritise?

Output numbered steps with the data structures and libraries (OR-Tools, DEAP,
NumPy) you intend to use.

Execute-phase prompt

What the analyst types to run it:

Execute the triage plan now. Walk data/corpus/, featurize every file into
items.csv (id, path, ext, size_bytes, cost_minutes, n_emails, n_iban, n_cards,
entropy, value_score) using the disclosed formulas. Then for a 960-minute
budget: run greedy, random (seed 42), and a 15-item brute-force; run the
OR-Tools 0/1 knapsack and assert value >= greedy, sweeping budgets
{120,240,480,720,960} to show where it strictly wins; run the CP-SAT
two-examiner split (2 x 480 min) with per-examiner load and value; run DEAP
NSGA-II and print a Pareto front of (value, types, minutes). Save to output.txt.
Do not fabricate any numbers — report exactly what the code produces.

The run: Approach → Action → Observation

Each Observation is verbatim from the real output.txt.

Step 1 — Featurize the corpus

Approach. I can't review 981 files in 16 h; first quantify a cost and a value for each file.

Action:  featurize("data/corpus")

Observation:
    files            : 981
    distinct types   : 22  -> {pdf:200, html:181, txt:154, doc:111, jpg:89, ppt:88, xls:62 ...}
    mean cost (min)  : 8.35   total : 8188.3 min (136.5 h to review ALL)
    files w/ PII hit : 349    wrote items.csv

Step 2 — Cheap baseline first

Approach. Establish a greedy (value/cost) baseline and a random control before optimising.

Action:  greedy(items, 960);  random(items, 960, seed=42)

Observation:
    greedy (value/cost):  90 files,  959.8 min, value = 26954.5
    random (seed=42)   : 112 files,  958.4 min, value =  4800.6

Step 3 — Provable optimum + where it beats greedy

Approach. Greedy is only a heuristic. Get the provable optimum and check it never loses; sweep budgets to expose where ordering matters.

Action:  ortools_knapsack(items, 960);  budget_sweep()

Observation:
    knapsack optimum : 90 files, 959.8 min, value = 26954.5  (>= greedy: PASS)
    budget sweep  greedy     knapsack    gain
       240 min    13758.5    13788.8    +30.2   <- knapsack WINS
       480 min    18804.4    18844.4    +40.0   <- knapsack WINS
       960 min    26954.5    26954.5     +0.0

Step 4 — Split work across two examiners

Approach. Two examiners are available; assign files to two 480-minute queues optimally.

Action:  cpsat_two_examiners(items, cap=480)

Observation:
    status = OPTIMAL
    examiner 1: 42 files, load 479.9/480 min, value = 16134.0
    examiner 2: 48 files, load 479.9/480 min, value = 10820.6  (total 26954.5)

Step 5 — Surface the trade-offs (Pareto front)

Approach. Value isn't the only goal — coverage and time matter too. Generate the non-dominated set.

Action:  nsga2_pareto(items, 960)

Observation:
    Pareto front size = 121; sample (value, types, minutes):
        value=14963.4  types=18  minutes=663.8
        value=12291.7  types=21  minutes=584.6
        value= 6417.3  types=15  minutes=257.1

>>> Human-in-the-loop gate. These are genuine trade-offs (more coverage costs value/time). The agent presents the Pareto options and halts — a qualified examiner picks the point (max value? max coverage? leave slack?). The optimiser frames the choice; the human owns the decision.

Done — the result

Optimised triage vs baselines (960 min budget):
    random    value  4800.6
    greedy    value 26954.5
    knapsack  value 26954.5   (optimal; strictly beats greedy at 240/480 min)
    -> ~5.6x the random baseline

Two examiners (2 x 480 min): OPTIMAL split, total value 26954.5
Pareto front: 121 defensible plans trading value / coverage / time

The headline for court: a ranked, budget-respecting examination order per examiner with a defensible value model, plus a Pareto front so the lead picks the trade-off that fits the case — not a single opaque "best" answer.

Guidance: reading the outcome

  • Greedy is strong but not optimal. At the full budget greedy ties the knapsack here, but the budget sweep proves the optimum strictly wins at tight budgets — exactly the regime where triage matters most.
  • The human picks the Pareto point. The agent halts after NSGA-II; an examiner chooses value vs coverage vs slack. The score frames the choice; the human owns the legal/operational call.
  • Failure mode — the value model is a proxy. A tiny image with no email/IBAN/card text (a scanned cheque, or a `.gz` that's really an encrypted archive scoring only on entropy) gets a low value and is deprioritised — the optimiser will confidently skip it. Mitigation: keep weights disclosed, always pair the plan with random spot-check sampling, and never treat "not selected" as "not evidence".

Honest deviation (teaching note). The documented subset0.zip path 404s; the run used the real govdocs1/zipfiles/000.zip (981 files). python-magic was unavailable, so file typing used extension + size. The knapsack==greedy tie at the full budget is a genuine result of this corpus's value distribution — the sweep is what surfaces the strict win.
