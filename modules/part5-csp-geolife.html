Constraint Satisfaction — the Agent's Run on Real GPS Data

This is the actual, executed outcome behind the worked prompt for constraint satisfaction. The pipeline really ran over a real Microsoft GeoLife trajectory; below is how an agent approaches it — Approach → Action → Observation → Done — with the genuine captured output woven into each observation. The data engineering (dataset, encoding, solver) lives on the Part 4 dataset page.

What was run

Forensic question: a suspect's phone places them at home (north Beijing) at 14:00 and 15:00; a CCTV hit reportedly places them at the crime scene (~67 km south) at 14:30; nobody travels faster than 120 km/h. Is that physically possible — and if not, which evidence conflicts? The agent encodes the hour as a constraint-satisfaction problem over a real GeoLife trajectory and solves it with OR-Tools CP-SAT. Verified outcome: the alibi is INFEASIBLE and the solver names the exact conflicting evidence; a re-timed sensitivity check is FEASIBLE.

Plan-phase prompt

What the analyst types to make the agent plan the analysis (no execution yet):

You are a digital-forensics analyst's planning assistant. Plan — do not yet
execute — a constraint-satisfaction analysis that tests whether a suspect's
alibi is physically consistent with the evidence.

Context:
  - Phone-provider records place the suspect at HOME (north Beijing) at 14:00
    and at 15:00.
  - A CCTV hit reportedly places them at the CRIME SCENE (south Beijing, ~67 km
    from home) at 14:30.
  - Assume a maximum travel speed of 120 km/h.

Produce a plan covering:
  1. Data: which open dataset grounds real coordinates (GeoLife GPS v1.3), how
     to download it, and the .plt format.
  2. Encoding: time-slot variables, region domains, and which evidence becomes
     which constraint (anchor / CCTV / reachability via a haversine matrix).
  3. Tooling: Python + OR-Tools CP-SAT; register CCTV and reachability as
     assumption literals so an UNSAT core can be extracted.
  4. Outputs: solver status for two scenarios, the minimal conflicting set on
     UNSAT, and a satisfying region-per-slot timeline on SAT.
  5. Reporting rule: the deliverable is "consistent" or "contradictory (with
     the conflicting evidence named)" — NEVER a guilt determination.

Return the plan as numbered steps with the exact tools/commands you will use.

Execute-phase prompt

What the analyst types to actually carry out the plan and run the tools:

Execute the plan now. Concretely:
  1. Create a venv with --system-site-packages and `pip install ortools`.
  2. Download GeoLife Trajectories 1.3 into data/ and unzip. If download fails,
     fall back to a format-accurate synthetic .plt, clearly labelled.
  3. Write and run run.py so that it: parses a real .plt (skip 6 header lines);
     builds the 4-region grid anchored on a real GPS point + haversine matrix;
     builds the CP-SAT model (7 slot vars, anchor + CCTV + reachability as
     assumption literals); solves scenario (a) CCTV @ CRIME_SCENE_SOUTH 14:30
     and on UNSAT prints the minimal conflict; solves scenario (b) with the CCTV
     retimed to a reachable region and prints the satisfying timeline.
  4. Capture the real output with `python3 run.py | tee output.txt`.
  5. Summarise consistency / contradiction — do not state guilt.

The run: Approach → Action → Observation

This is the agent's reasoning loop. Each Observation below is verbatim from the real output.txt — no edited numbers.

Step 1 — Ground the geography in real data

Approach. I need real Beijing coordinates so the distances are credible; parse a GeoLife trajectory.

Action:  parse_plt("data/.../010/Trajectory/20070804033032.plt")

Observation:
    Parsed 1116 GPS points (6 header lines skipped).
    Sample row 1: lat=39.921712, lon=116.472343, alt=13.0, date=2007-08-04, time=03:30:32

Step 2 — Build the region grid and distance matrix

Approach. Bin space into regions around the anchor and compute how far apart they are, so I know what is reachable in one 10-minute slot (max 20 km at 120 km/h).

Action:  build_regions(anchor); haversine_distance_matrix()

Observation:
    Haversine distance matrix (km), max reach per 10 min = 20 km:
                        HOME_NOR   MIDTOWN  CRIME_SC  ELSEWHER
    HOME_NORTH               0.0      16.7      66.7      61.0
    MIDTOWN                 16.7       0.0      50.0      53.8
    CRIME_SCENE_SOUTH       66.7      50.0       0.0      61.2
    -> CRIME_SCENE_SOUTH is 66.7 km from HOME: NOT reachable in one slot.

Step 3 — Encode evidence as a CSP and solve (scenario a)

Approach. Anchor HOME at 14:00 and 15:00 (hard); add the CCTV claim and each reachability hop as assumption literals; solve with CP-SAT.

Action:  solve_alibi(cctv = CRIME_SCENE_SOUTH @ 14:30)        [CP-SAT]

Observation:
    Solver status : INFEASIBLE
    -- the evidence is mutually contradictory.

Step 4 — Extract the minimal conflict (UNSAT core)

Approach. Infeasible means the evidence clashes; extract the smallest conflicting set so I can name which pieces of evidence cannot co-exist.

Action:  solver.SufficientAssumptionsForInfeasibility()

Observation:
    Minimal conflicting constraint set (UNSAT core):
       - CCTV: suspect at CRIME_SCENE_SOUTH at 14:30
       - Reachability 14:00->14:10 (<= 20 km in 10 min)
       - Reachability 14:10->14:20 (<= 20 km in 10 min)
       - Reachability 14:20->14:30 (<= 20 km in 10 min)

>>> Human-in-the-loop gate. The agent stops and reports a contradiction. It does not — must not — output a guilt verdict. The analyst decides next steps (is the CCTV timestamp reliable? re-examine the phone record?).

Step 5 — Sensitivity check (scenario b)

Approach. What if the CCTV time/place is revised to a reachable region (MIDTOWN, 16.7 km from home)? Re-solve.

Action:  solve_alibi(cctv = MIDTOWN @ 14:30)                  [CP-SAT]

Observation:
    Solver status : OPTIMAL  (FEASIBLE)
    Consistent timeline:
       14:00 -> HOME_NORTH   14:10 -> MIDTOWN   14:20 -> HOME_NORTH
       14:30 -> MIDTOWN      14:40 -> HOME_NORTH 14:50 -> HOME_NORTH  15:00 -> HOME_NORTH

Done — the result

Scenario (a): INFEASIBLE — alibi contradicted. The phone-at-HOME anchors and the CCTV hit at CRIME_SCENE_SOUTH are physically irreconcilable at 120 km/h. The solver returns the minimal conflicting evidence set {CCTV@14:30, reachability 14:00→14:10, 14:10→14:20, 14:20→14:30} — a precise, court-ready statement of what clashes.

Scenario (b): FEASIBLE. Re-timing the CCTV hit to the reachable MIDTOWN district yields a fully consistent HOME↔MIDTOWN timeline — proving the method does not simply always cry "contradiction". The deliverable is a concrete assignment (or a named conflict), never a probability or a verdict.

Guidance: reading the outcome

  • Two terminal modes. UNSAT → the evidence is contradictory; report the named conflict. SAT → at least one consistent timeline exists; report it. Neither is a statement of guilt.
  • The gate fires on UNSAT. The agent surfaces the minimal conflicting set and hands control back — it never escalates to a conclusion.
  • Failure mode. The result is only as good as the inputs: a wrong CCTV timestamp, a spoofed phone location, or a too-coarse region grid can flip SAT/UNSAT. Treat the conflict set as a lead to verify the inputs, not as proof.

Honest deviation (teaching note). The GPS trajectory is real; the criminal framing (suspect, CCTV, crime scene) is a constructed teaching scenario. Scenario (b) uses an in-window reachable district rather than the "15:30" idea from the brief, because the modelled hour ends at 15:00.
